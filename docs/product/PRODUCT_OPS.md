# Product Operations Playbook

**What this doc is:** The end-to-end operational reference for how work moves through Dugout Lineup — from idea to shipped. Covers story creation, labeling, branching, PR conventions, fix promotion, and the release ritual.

**Related docs:**
- `docs/process/ISSUE_TRACKING.md` — label taxonomy, scripts, commit/PR conventions (technical mechanics)
- `CLAUDE.md` — ship gate, pre-release checklist, architectural rules
- `docs/product/ROADMAP.md` — the live backlog and release history
- `docs/product/MASTER_DEV_REFERENCE.md` — infra identifiers, incident response, key file locations

## The Operational Loop (overview)
Idea / Bug discovered
↓
Story written in ROADMAP.md
↓
GitHub Issue filed (same session) → labeled → linked via <!-- #N -->
↓
Feature branch cut from develop
↓
Code written → tests RED → GREEN → docs updated
↓
PR opened → develop (squash merge)
↓
Overnight soak on the Vercel preview URL for the PR branch (auto-generated by Vercel on every PR push)
↓
Ship Gate + Pre-release Docs Checklist passed
↓
PR opened → main (squash merge)
↓
Production deploy (Vercel auto, Render auto)
↓
GitHub Issue auto-closed via closes #N
↓
Story marked ✅ Resolved in ROADMAP.md

## 1. Capturing a Story

### Where stories live
Every piece of work — bug, feature, chore, governance — starts as a Story in `docs/product/ROADMAP.md` and a matching GitHub Issue. These are the two sides of the same coin: ROADMAP is the human-readable narrative, GitHub Issues is the trackable system of record.

### Story template (ROADMAP.md)
Story [N] (P[0-3]) — [short name] <!-- #GH_ISSUE_NUMBER -->
Status: Open | In Progress | Resolved | Deferred
Discovered: [date + context]
Target: [version]
Symptom: [what goes wrong]
Impact: [who's affected, how bad]
Root cause: [known | hypothesis | unknown]
Proposed fixes: [options with tradeoffs]
Recommendation: [which option + why]

### Priority rules
| Label | When to use |
|---|---|
| `priority:p0` | Production broken, data at risk, auth principle violated — drop everything |
| `priority:p1` | Blocks a coach at the field, no workaround — fix this release |
| `priority:p2` | Important but not blocking — next 1–2 releases |
| `priority:p3` | Nice to have, quality, observability — backlog |

### Hygiene rules (non-negotiable)
1. Every Story in ROADMAP.md must have a GitHub Issue filed the same session
2. Use the **📋 Story** template at `github.com/kaushikkuberanathan/lineup_generator/issues/new/choose`
3. Add the issue number to the Story heading: `<!-- #132 -->`
4. Apply at minimum: one `priority:*` label + one `type:*` label
5. Run `node scripts/sync-stories-to-issues.js --dry-run` for batch syncs — review before live run

## 2. Labeling

All issues use `prefix:name` convention (no spaces). Full taxonomy in `docs/process/ISSUE_TRACKING.md`.

### Minimum required on every issue
- `priority:p0` / `p1` / `p2` / `p3`
- `type:bug` / `type:feature` / `type:chore` / `type:governance`

### Recommended (add when known)
- `area:*` — which part of the system (scoring, auth, ux, backend, ci-ops, share-link, roster, supabase, analytics)
- `status:in-progress` — when work starts
- `status:deferred` — when explicitly pushed to a future phase

### Auto-applied labels
- `auto-created` — added by scripts and health check automation
- `source:coach-feedback` — added when a bug originates from in-app feedback
- `needs-overnight-soak` — add when merging to develop, remove when soak passes
- `hotfix-exception` — add when bypassing Ship Gate; reason must be in commit message

## 3. Branch Workflow

### Branch naming
| Type | Pattern | Example |
|---|---|---|
| Feature | `feature/[short-name]` | `feature/combined-game-view` |
| Bug fix | `fix/[story-or-description]` | `fix/share-link-routing` |
| Chore | `chore/[description]` | `chore/sync-stories-to-issues` |
| Hotfix | `hotfix/[description]` | `hotfix/share-link-viewer-routing` |
| Docs | `docs/[description]` | `docs/issue-tracking-hygiene` |

### Branch rules
- **Always** cut feature/fix branches from `develop`, not `main`
- **Never** commit directly to `develop` or `main` (Husky pre-push hook enforces this)
- Emergency override: `$env:ALLOW_DIRECT_PUSH=1; git push origin develop --no-verify` — requires justification in commit message
- Post-squash cleanup: `git branch -D [branch]` (capital D — squash merges mean local branch isn't "merged" by Git's definition)

### Worktree pattern
KK operates two worktrees in parallel:
- `lineup-generator` — main worktree, `develop` branch, primary development
- `lineup-generator-ux` — UX worktree, active UX feature branch

When develop advances, sync the UX worktree:
```powershell
cd C:\Users\KKUBERANA1\Documents\lineup-generator-ux
git fetch origin
git merge origin/develop --no-edit
```

## 4. Development Loop

### Before writing code
1. Confirm the Story has a GitHub Issue with `<!-- #N -->` in ROADMAP.md
2. Cut a feature branch from `develop`
3. Write a failing test (RED) before touching implementation code

### While coding
- One step per Claude Code message — never queue multiple steps
- Stage specific paths only: `git add <specific-path>` — never `git add -A`
- Commit messages reference the issue: `fixes #127` or `closes #127`
- Certain files require a gate phrase before editing — `frontend/src/App.jsx`, `frontend/src/utils/migrations.js`, `frontend/src/utils/formatters.js`, `frontend/src/utils/flagBootstrap.js`, `frontend/src/components/game-mode/*`, `frontend/src/components/ScoringMode/*`, `frontend/package.json`, `backend/package.json`, `CLAUDE.md`. Gate phrase: *"all clear — App.jsx editing approved"* (or file-specific equivalent). See `CLAUDE.md` for current gate phrases.

### Before committing
- Run full test suite: `cd frontend && npm test` — all tests must pass
- Confirm no unrelated files staged: `git status`
- Write commit message via temp file (Windows UTF-8 BOM workaround):
```powershell
[System.IO.File]::WriteAllText("commit_msg.txt", "type: description`n`nbody", [System.Text.UTF8Encoding]::new($false))
git commit -F commit_msg.txt
Remove-Item commit_msg.txt
```

## 5. Pull Requests

### Every PR requires
- **Title:** `type: short description` (e.g. `fix: share link routing`, `feat: combined game view`)
- **Body:** fully filled PR template (never partial) — see `.github/pull_request_template.md`
- **Related Issue:** `closes #N` for every story resolved by this PR
- **Type of change:** exactly one checkbox checked
- **Pre-release Docs Checklist:** all relevant items checked or marked N/A
- **Ship Gate:** all four questions answered or marked exempt with reason

### Merge rules
| Scenario | Merge type |
|---|---|
| Feature/fix/chore/docs branch → develop | **Squash merge** (one clean commit per PR) |
| develop → main (release) | **Squash merge** |
| main → develop (post-release sync) | **Regular merge** (preserves merge commit on develop) |

**Never squash a sync PR** (main → develop). The merge commit is required so develop's ancestry stays intact for future conflict resolution.

### PR smoke test (before opening)
- [ ] `npm run build` clean
- [ ] Full Vitest suite passing
- [ ] Vercel preview deployed
- [ ] Phone smoke test on preview URL (not DevTools simulation)

## 6. Fix Promotion (develop → main)

### Overnight soak — non-negotiable
Every PR to `main` requires at least one overnight cycle on the Vercel preview URL for the PR branch (auto-generated by Vercel on every PR push) first. No exceptions. Not overridable by game-day pressure.

Exempt: docs-only meta-governance patches and declared hotfixes (must include `[hotfix-exception]` in commit message).

### Ship Gate (four questions, answer before every release)
1. Does every feature touched have a test covering the golden path?
2. Does every touched feature have docs reflecting current behavior?
3. Does `FEATURE_MAP.md` have a current row for every touched feature?
4. Are all P0 items in `DOC_TEST_DEBT.md` resolved or explicitly unblocked?

If any answer is "no" → stop. Document the debt, then decide whether to proceed.

### Pre-release Docs Checklist (16 items, walk before every develop → main PR)
Full list in `CLAUDE.md § Pre-release Docs Checklist`. Key items:
- `APP_VERSION` bumped in `App.jsx`
- Both `package.json` files bumped
- `VERSION_HISTORY` entry prepended (userChanges + internalChanges + techNote)
- `ROADMAP.md` release entry added, completed stories moved to shipped
- `FEATURE_MAP.md` rows updated for every touched feature
- Vercel preview phone-smoke-tested on real device + network

### Version bump ritual
Run once per release (replace values as needed):
```powershell
# 1. Bump APP_VERSION in frontend/src/App.jsx
# 2. Bump version in frontend/package.json
# 3. Bump version in backend/package.json
# 4. Add VERSION_HISTORY entry (userChanges must answer "what does coach experience differently?")
# 5. Add ROADMAP.md release entry
# 6. Update CLAUDE.md current version line
```

Future: `scripts/bump-version.ps1` will automate steps 1–3 (not yet built — Story backlog).

### After merging to main
1. Verify Vercel deploy succeeded at `dugoutlineup.com`
2. Verify `/ping` returns 200 in <2s
3. Open share link on mobile — confirm unauthenticated viewer loads
4. Close the GitHub Issues resolved by this release (auto-closed via `closes #N` if used correctly)
5. Mark Stories `✅ Resolved vX.X.X` in ROADMAP.md
6. Sync develop: open a `main → develop` regular merge PR

## 7. Hotfix Path

For P0 production incidents only. Bypasses the overnight soak requirement.

```powershell
# 1. Cut from main (not develop)
git checkout main
git checkout -b hotfix/[description]

# 2. Fix + tests (RED → GREEN still required)
# 3. Commit with hotfix marker
# Commit message body must include: [hotfix-exception] reason: [one sentence]

# 4. PR directly to main
# 5. After merge, immediately sync to develop
git checkout develop
git merge main --no-edit
git push origin develop
```

Apply `hotfix-exception` label on the GitHub Issue. Document the bypass in the commit message — it self-audits in `git log`.

## 8. Closing the Loop

When a story ships:
1. GitHub Issue auto-closes via `closes #N` in the merge commit (if used correctly)
2. If not auto-closed: close manually and add a comment with the version that shipped it
3. Update ROADMAP.md Story heading: `### ✅ Story [N] (P[X]) — [name] <!-- #N -->`
4. Add `Status: Resolved vX.X.X` to the Story body
5. Run the audit cadence check (every other session):
   - Open `DOC_TEST_DEBT.md` — any P0 still open? Any newly resolved?
   - Did this session change any feature's behavior? → Update `FEATURE_MAP.md`
